# Бронирование поездок. Чаты.

Подробная модель сообщений чата и миграция `content -> payload (jsonb)` описаны в `docs/features/chat-messages.md`.

Эндпоинты поездок (`/api/trip/...`) и правило **аутентификации** для `GET /api/trip/owned` см. в `docs/features/trips.md`.

Документ закрепляет контракт для **фронтенда** (в т.ч. обзор водителя) и для бекенда; детали полей — в **OpenAPI** / фактических ответах API.

## Статусы бронирования

1. WAITING — заявка создана, ждёт решения
1. ACCEPTED — подтверждено владельцем (актуально для логики “видимости старых поездок” в коде)
1. REJECTED — отклонено владельцем
1. CANCELLED — отменено (любой стороной в рамках ваших правил)
1. EXPIRED — протухло/не подтверждено вовремя
1. NO_SHOW — неявка пассажира
1. CHECKED_IN — отметка/посадка зафиксирована
1. IN_PROGRESS — поездка началась
1. COMPLETED — завершено

### «Занятое» пассажирское место (ёмкость `trips.passengers`)

В одном источнике правды (см. `BookingStatus` и `BookingRepository.countByTripIdAndBookingStatusIn`) для **лимита мест** и для **сводки** в обзоре водителя учитываются бронирования в статусах:

- **ACCEPTED**
- **CHECKED_IN**
- **IN_PROGRESS**

Остальные статусы (`WAITING`, `REJECTED`, `CANCELLED`, `EXPIRED`, `NO_SHOW`, `COMPLETED`) к подсчёту занятых мест **не** относятся.

При создании новой заявки и при переходе **`WAITING → ACCEPTED`** проверяется, что число таких бронирований **меньше** `TripEntity.passengers` (ёмкость пассажирских мест в машине).

### Терминальные статусы (фильтр обзора поездки)

Для **`GET /api/booking/trip/{tripId}/overview`** и параметра **`bookingScope`** «архивность» строки считается **терминальной**, если статус:

`COMPLETED`, `REJECTED`, `CANCELLED`, `EXPIRED`, `NO_SHOW`.

### Параметр `bookingScope` (таблица)

Правило применяется **только к строкам** в **`bookings.content`**; объект **`summary`** от фильтра **не** зависит (считается по **всем** бронированиям поездки).

| Значение | Строка попадает в `bookings.content`, если |
|----------|---------------------------------------------|
| **`active`** | `trip.start >= now` **и** статус бронирования **не** терминальный |
| **`archived`** | `trip.start < now` **или** статус терминальный |
| **`all`** | фильтр по этому правилу не применяется |

Некорректное значение `bookingScope` → **400 BAD_REQUEST**.

### Пагинация: `owned` vs `overview`

- **`GET /api/trip/owned`** (`trips.md`) — поддерживается Spring **`Pageable`** (`page`, `size`, `sort`).
- **`GET .../overview`** — **без** `page`/`size` в query; список отдаётся **целиком** за один запрос. В JSON поле **`bookings`** всё равно сериализовано как **`Page`**, но с **`Pageable.unpaged()`** (обычно `totalElements` = числу элементов в `content`).

## Переходы статусов

- `WAITING -> ACCEPTED` — владелец поездки подтвердил заявку.
- `WAITING -> REJECTED` — владелец поездки отклонил заявку.
- `WAITING -> CANCELLED` — заявка отменена пассажиром или владельцем до принятия решения.
- `WAITING -> EXPIRED` — заявка не была обработана вовремя.

- `ACCEPTED -> CANCELLED` — подтверждённое бронирование отменено одной из сторон.
- `ACCEPTED -> NO_SHOW` — пассажир не явился к согласованному времени/точке встречи.
- `ACCEPTED -> CHECKED_IN` — посадка/прибытие пассажира подтверждены.

- `CHECKED_IN -> CANCELLED` — поездка сорвалась после фиксации посадки, но до старта.
- `CHECKED_IN -> IN_PROGRESS` — поездка фактически началась.

- `IN_PROGRESS -> COMPLETED` — поездка завершена успешно.

## Кто меняет статус

- Переход в **`ACCEPTED`** (из `WAITING`) — владелец поездки.
- Переход в **`REJECTED`** (из `WAITING`) — владелец поездки.
- `EXPIRED` — система по таймауту или владелец поездки через служебный сценарий.
- `NO_SHOW` — владелец поездки.
- `IN_PROGRESS` — владелец поездки.
- `COMPLETED` — владелец поездки.
- `CHECKED_IN` — пассажир как основное действие. Если потребуется более строгая схема, можно заменить на двухстороннее подтверждение.
- `CANCELLED` — любая сторона.

## Правила по событиям

- `EXPIRED`:
  заявка оставалась в `WAITING` и не была принята/отклонена к моменту старта поездки (`trip.start`).
  Важно: **из `ACCEPTED` в `EXPIRED` перехода нет**.
- `NO_SHOW`:
  пассажир не прибыл в согласованное место/время, хотя бронирование было `ACCEPTED`.
- `CHECKED_IN`:
  пассажир прибыл, посадка или готовность к посадке зафиксирована.
- `IN_PROGRESS`:
  машина поехала, поездка началась фактически.
- `COMPLETED`:
  поездка завершена. Статус может проставляться **позже**, когда продукт предлагает оставить отзыв о попутчике.

## Чат бронирования

- Чат создаётся в момент создания бронирования, то есть при статусе `WAITING`.
- Чат остаётся доступным для обычных сообщений и после завершения поездки: людям может понадобиться обсудить детали/договориться о возврате вещей и т.п.
- UI-ограничения (какие кнопки действий показывать) зависят от текущего `booking_status` и роли пользователя, но это не блокирует обычный текстовый чат.

## Сервисные сообщения о смене статуса

- Любая смена статуса бронирования, инициированная из чата, должна сопровождаться записью в `message` с `payload`:
  - `type = "SERVICE"`
  - `event = "BOOKING_STATUS_CHANGED"`
  - `from`, `to` — предыдущий и новый статус
- Поле `action` в payload **не используется**: достаточно `from/to`.

## API

Маршрутизация: префикс `/trip/` отделяет **id поездки** от путей, где первый сегмент — **id бронирования** (UUID), поэтому важен порядок объявления `@GetMapping` (например, `/trip/{tripId}/overview` не конфликтует с `{bookingId}`).

### Поездка: краткий список бронирований

- **`GET /api/booking/trip/{tripId}`**
  - **Auth**: авторизованный **владелец** поездки.
  - **Ответ**: `List<BookingDto>` (компактно: идентификатор бронирования, `tripId`, `passengerId`, `bookingStatus`).
  - Если пользователь не владелец (или поездка не найдена) — **`404 NOT_FOUND`** (маскирование доступа совпадает с обзором).

### Поездка: обзор для водителя (сводка мест + диалоги)

- **`GET /api/booking/trip/{tripId}/overview`**
  - **Auth**: только **владелец** поездки; иначе **`404 NOT_FOUND`**.
  - **Query**: `bookingScope` (необязательно, по умолчанию `all`):
    - **`all`** — без фильтра по правилу ниже.
    - **`active`**: одновременно `trip.start >= now` **и** статус бронирования **не** терминальный (см. список выше).
    - **`archived`**: `trip.start < now` **или** статус терминальный.
    - недопустимое значение — **`400 BAD_REQUEST`**.
  - **Ответ**: `TripBookingOverviewResponseDto`:
    - **`summary`** — счётчики **по всей поездке**, **не зависят** от `bookingScope`: `tripId`, `passengerSeatCapacity`, `occupiedSeats`, `freePassengerSeats`, `full` (`occupiedSeats` считается по статусам «занятое место», см. выше).
    - **`bookings`** — Spring Data **`Page<TripBookingOverviewItemDto>`** в режиме **`Pageable.unpaged()`** (за один запрос отдаются **все** строки, попавшие под `bookingScope`). В JSON элементы находятся в **`bookings.content`**; полезные для UI метаданные: **`bookings.totalElements`**. При стандартной сериализации Jackson могут присутствовать и «служебные» ключи страницы (`pageable`, `sort`, …); при необходимости контракт можно упростить на бекенде отдельной настройкой Jackson.
    - Поле **`version` поездки** в этом ответе **не возвращается** (текущие договорённости).
  - **`TripBookingOverviewItemDto`**: `bookingId`, `bookingStatus`, **`passengerDisplayName`** (только ФИО из профиля), превью последнего сообщения (`lastMessagePayload`, `messageStatus`, `lastMessageTime`). Если сообщений в чате ещё нет, часть полей превью может быть **`null`**. **`username`/email/Telegram` не возвращаются**.
  - Сообщения чата построчно: по-прежнему **`GET /api/booking/messages/{bookingId}`**.

Старый **`GET /api/booking/trip/{tripId}`** и новый **`/overview`** **сохраняются параллельно** (разный уровень детализации).

### Связанные эндпоинты и коды ошибок

| Метод | Путь | Кратко |
|--------|------|--------|
| `GET` | `/api/booking/trip/{tripId}` | Компактный список владельца. Не владелец / нет поездки → **404**. |
| `GET` | `/api/booking/messages/{bookingId}` | История чата; не участник → **403**. |
| `POST` | `/api/booking/{bookingId}/status` | Смена статуса; не участник → **403**; нет мест при `WAITING → ACCEPTED` → **400**. |
| `GET` | `/api/trip/{id}` | Карточка поездки; правила «старых» поездок — в `trips.md`. |

### Прочее

- **`GET /api/booking/{bookingId}/available-statuses`** — допустимые переходы статуса для текущего пользователя (`bookingId` — UUID строки бронирования). Не участник → **403**.
- **`PUT /api/booking`**, общий **`GET /api/booking/`** (списки чатов) и остальное — см. **`BookingController`**.

Подробнее по кодам ответов для сообщений и смены статуса см. таблицу «Связанные эндпоинты» выше.

## Рекомендации для фронтенда

- Значения «занято / свободно / полная машина» брать только из **`summary`** в **`GET .../overview`**, не пересчитывать по отфильтрованному **`bookings.content`**.
- Список строк под **`bookingScope`** приходит **одним** запросом; дополнительную группировку/сортировку можно делать на клиенте.
- По **`GET /api/trip/owned`** использовать **`content`** страницы `Page<TripDto>` как основной массив списка; «все роли» — **не передавать** `participant`, водитель — **`participant=owner`**, пассажир — **`participant=passenger`** (см. `trips.md`).

## Завершённые шаги миграции чата

- Ответ сообщений содержит `payload` из `jsonb`; при смене статуса создаются `SERVICE` сообщения (`BOOKING_STATUS_CHANGED`).

## Примечания по реализации

- `ACCEPTED` остаётся важным статусом для текущей логики доступа к старым поездкам.
- Если в будущем понадобится точнее различать отмены, лучше не раздувать enum, а добавить отдельные поля вроде `cancelledBy` и `cancelReason`.
- Если потребуется двусторонняя фиксация начала/завершения поездки, это лучше делать отдельными событиями, а не дополнительными статусами.
