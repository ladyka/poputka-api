# Бронирование поездок. Чаты.

Подробная модель сообщений чата и миграция `content -> payload (jsonb)` описаны в `docs/features/chat-messages.md`.

## Статусы бронирования

1. WAITING — заявка создана, ждёт решения
1. ACCEPTED — подтверждено владельцем (актуально для логики “видимости старых поездок” в коде)
1. REJECTED — отклонено владельцем
1. CANCELLED — отменено (любой стороной в рамках ваших правил)
1. EXPIRED — протухло/не подтверждено вовремя
1. NO_SHOW — неявка пассажира
1. CHECKED_IN — отметка/посадка зафиксирована
1. IN_PROGRESS — поездка началась
1. COMPLETED — завершено

## Переходы статусов

- `WAITING -> ACCEPTED` — владелец поездки подтвердил заявку.
- `WAITING -> REJECTED` — владелец поездки отклонил заявку.
- `WAITING -> CANCELLED` — заявка отменена пассажиром или владельцем до принятия решения.
- `WAITING -> EXPIRED` — заявка не была обработана вовремя.

- `ACCEPTED -> CANCELLED` — подтверждённое бронирование отменено одной из сторон.
- `ACCEPTED -> NO_SHOW` — пассажир не явился к согласованному времени/точке встречи.
- `ACCEPTED -> CHECKED_IN` — посадка/прибытие пассажира подтверждены.

- `CHECKED_IN -> CANCELLED` — поездка сорвалась после фиксации посадки, но до старта.
- `CHECKED_IN -> IN_PROGRESS` — поездка фактически началась.

- `IN_PROGRESS -> COMPLETED` — поездка завершена успешно.

## Кто меняет статус

- `ACCEPTED` — владелец поездки.
- `REJECTED` — владелец поездки.
- `EXPIRED` — система по таймауту или владелец поездки через служебный сценарий.
- `NO_SHOW` — владелец поездки.
- `IN_PROGRESS` — владелец поездки.
- `COMPLETED` — владелец поездки.
- `CHECKED_IN` — пассажир как основное действие. Если потребуется более строгая схема, можно заменить на двухстороннее подтверждение.
- `CANCELLED` — любая сторона.

## Правила по событиям

- `EXPIRED`:
  заявка оставалась в `WAITING` и не была принята/отклонена к моменту старта поездки (`trip.start`).
  Важно: **из `ACCEPTED` в `EXPIRED` перехода нет**.
- `NO_SHOW`:
  пассажир не прибыл в согласованное место/время, хотя бронирование было `ACCEPTED`.
- `CHECKED_IN`:
  пассажир прибыл, посадка или готовность к посадке зафиксирована.
- `IN_PROGRESS`:
  машина поехала, поездка началась фактически.
- `COMPLETED`:
  поездка завершена. Статус может проставляться **позже**, когда продукт предлагает оставить отзыв о попутчике.

## Чат бронирования

- Чат создаётся в момент создания бронирования, то есть при статусе `WAITING`.
- Чат остаётся доступным для обычных сообщений и после завершения поездки: людям может понадобиться обсудить детали/договориться о возврате вещей и т.п.
- UI-ограничения (какие кнопки действий показывать) зависят от текущего `booking_status` и роли пользователя, но это не блокирует обычный текстовый чат.

## Сервисные сообщения о смене статуса

- Любая смена статуса бронирования, инициированная из чата, должна сопровождаться записью в `message` с `payload`:
  - `type = "SERVICE"`
  - `event = "BOOKING_STATUS_CHANGED"`
  - `from`, `to` — предыдущий и новый статус
- Поле `action` в payload **не используется**: достаточно `from/to`.

## API

- `GET /api/booking/trip/{tripId}` — список бронирований поездки (владелец поездки).
- `GET /api/booking/{bookingId}/available-statuses` — какие переходы доступны текущему пользователю (`bookingId` — UUID бронирования).
- `POST /api/booking/{bookingId}/status` — смена статуса из чата (тело: `{ "to": "ACCEPTED" }`), сервер создаёт сервисное сообщение.

Маршрутизация: префикс `/trip/` отделяет **id поездки** от путей, где первый сегмент — **id бронирования** (UUID), поэтому в шаблонах `@GetMapping` не нужны регекспы на path variables.

## План изменений в коде (после миграции)

- `BookingMessage` / `MessageService` / DTO для сообщений: перейти с `content` на `payload` (`jsonb` в БД).
- Endpoint получения сообщений: вместо “строки контента” отдавать JSON, полученный из `payload` (как минимум поле `payload` в ответе).
- Добавить создание `SERVICE` сообщений при смене статуса.

## Примечания по реализации

- `ACCEPTED` остаётся важным статусом для текущей логики доступа к старым поездкам.
- Если в будущем понадобится точнее различать отмены, лучше не раздувать enum, а добавить отдельные поля вроде `cancelledBy` и `cancelReason`.
- Если потребуется двусторонняя фиксация начала/завершения поездки, это лучше делать отдельными событиями, а не дополнительными статусами.
