# Техническая документация: модуль взаимопомощи на дороге

Документ описывает модуль помощи на дороге (roadassistance) в приложении-попутке.  
Фокус — на том, что действительно используется в коде (контроллеры, статусы, ограничения), без лишних SQL‑деталей.

## 1. Обзор и цели

- **Назначение**: пользователи могут запрашивать помощь на дороге и оказывать её другим.
- **Основные сценарии**:
    - создать запрос о помощи с геопозицией и описанием проблемы;
    - предложить помощь по существующему запросу;
    - завершить помощь и оставить отзыв;
    - настроить профиль «помощника» (готовность помогать, радиус, компетенции);
    - модерация пользователей с низким рейтингом.

Базовый путь API модуля: `/api/roadassistance`.

## 2. Бизнес-требования (кратко)

### 2.1. Типы помощи

Примеры типов:
- буксировка / тягач;
- прикурить / отогреть;
- доставить топливо;
- замена колеса;
- незначительный ремонт;
- вскрыть авто;
- просто подождать вместе;
- другое.

### 2.2. Ключевые принципы

- Любой **авторизованный** пользователь может создать публичный запрос о помощи.
- Все пользователи видят **активные** запросы по текущим статусам (`CREATED`, `OFFER_RECEIVED`). Фильтрация по `expires_at` и точный учёт георадиуса реализованы частично: истечение переводит запрос в `EXPIRED` через `processExpiredRequests`, но запросы в `findByStatusInAndLocationNear` пока не ограничиваются по `expires_at`.
- Пользователи с флагом `readyToHelp = true` получают уведомления о новых запросах; в текущем коде отправка выполняется всем готовым помощникам (учёт `helpRadius` пока `TODO`).
- Помощники настраивают компетенции/инструменты в профиле.
- Есть внутренняя система чатов и уведомлений (переиспользуется существующая инфраструктура).
- Ведётся рейтинг и отзывы, пользователи с рейтингом < 3.0 попадают на модерацию.

## 3. Модель данных (логический уровень)

Ниже — логическое описание сущностей. Точные определения колонок заданы в Liquibase‑миграциях в `db/changelog`.

- **`assistance_types`** — справочник типов помощи.
    - `code`, `name`.
- **`competencies`** — справочник компетенций/инструментов.
    - может быть привязан к `assistance_types`.
- **`assistance_requests`** — запросы о помощи.
    - автор (`author_id`), тип проблемы (`problem_type_id`), координаты (lat/lon), адрес;
    - статус: `CREATED`, `OFFER_RECEIVED`, `OFFER_ACCEPTED`, `IN_PROGRESS`, `COMPLETED`, `CANCELLED`, `EXPIRED`, `FAILED`;
    - времена: `expires_at`, `initial_expires_at`, `max_expires_at`, `last_activity_at`.
- **`assistance_offers`** — предложения помощи.
    - ссылка на запрос (`request_id`), помощника (`helper_id`);
    - статус: `PENDING`, `ACCEPTED`, `REJECTED`, `CANCELLED`;
    - `chat_id` — связанный чат в существующей системе.
- **`assistance_reviews`** — отзывы.
    - `request_id`, `reviewer_id`, `reviewee_id`, `rating (1..5)`, `comment`;
    - `is_auto_generated`, `is_moderated`, `moderator_comment`.
- **`users` (расширения)**:
    - `readyToHelp`, `helpRadius`, `rating`, `requiresModeration`, `isBlocked`.
- **`user_competencies`** — связь пользователей и компетенций.

## 4. Основные бизнес-процессы

### 4.1. Создание запроса о помощи

1. Валидация: пользователь не заблокирован (`isBlocked = false`), при необходимости — рейтинг ≥ 3.0.
2. Создание `assistance_requests`:
    - `status = CREATED`;
    - `expires_at = now + 2 часа`;
    - `max_expires_at = now + 48 часов`;
    - `initial_expires_at = expires_at`;
    - `last_activity_at = now`.
3. Поиск помощников в радиусе (`helpRadius`) с подходящими компетенциями.
   - В текущем коде поиск/геофильтрация помечены `TODO`, уведомление отправляется всем `readyToHelp`.
4. Отправка уведомлений потенциальным помощникам.

### 4.2. Продление запроса

Условия:
- только автор запроса;
- статус ∈ {`CREATED`, `OFFER_RECEIVED`};
- новое `expires_at` не превышает `max_expires_at`.

Действия:
- `expires_at = min(expires_at + выбранный_интервал, max_expires_at)`;
- обновление `last_activity_at`;
- уведомление участников чатов запроса.

### 4.3. Предложение помощи и автоотзыв

- Помощник может создать предложение для запроса.
- При создании:
    - создаётся запись в `assistance_offers` со статусом `PENDING`;
    - создаётся чат типа `ASSISTANCE`, `chat_id` сохраняется в предложении.
- При отмене предложения **после 15 минут** предусмотрен автоматический отзыв. В текущем коде:
    - отмена предложения доступна только при статусе предложения `PENDING`;
    - условие по 15 минутам в `AssistanceOfferService.cancelOffer` есть, но создание auto-review помечено `TODO`/закомментировано (в закомментированном коде фигурировал рейтинг `2`).
- После сохранения **любого** отзыва пересчитывается рейтинг пользователя.

### 4.4. Рейтинг и модерация

- Базовый рейтинг пользователя — среднее арифметическое последних 50 валидных отзывов.
- Для расчёта берутся только:
    - ручные отзывы (`is_auto_generated = false`) или
    - отзывы, прошедшие модерацию (`is_moderated = true`).
- При падении рейтинга ниже 3.0:
    - `requiresModeration = true` и отправляется уведомление модераторам (`LOW_RATED_USER`);
    - фактическая блокировка создания запросов и откликов реализована через проверку `rating >= 3.0` в `createRequest`/`createOffer`;
    - дополнительная блокировка по `requiresModeration` на уровне сервисов пока не реализована полностью (`TODO`).

## 5. API модуля roadassistance

Базовый путь: `/api/roadassistance`.

### 5.1. Реализованные эндпоинты

#### 5.1.1. Запросы о помощи (`AssistanceRequestController`)

- `GET /api/roadassistance/requests`  
  Поиск активных запросов с фильтрацией по координатам (`lat`, `lon`, `radiusKm`) и пагинацией.
- `GET /api/roadassistance/requests/{id}`  
  Детали запроса (`404`, если запроса нет).
- `POST /api/roadassistance/requests`  
  Создать запрос о помощи.  
  Вход: параметры авто, тип проблемы (`problemTypeCode`), описание, координаты и адрес.  
  Требует авторизованного пользователя.
- `POST /api/roadassistance/requests/{id}/extend`  
  Продлить срок действия запроса (только автор, с проверкой ограничений по статусу и `max_expires_at`).
- `POST /api/roadassistance/requests/{id}/cancel`  
  Отменить запрос (только автор).
- `GET /api/roadassistance/requests/active/count`  
  Количество активных запросов в заданном радиусе относительно координат.

#### 5.1.2. Предложения помощи (`AssistanceOfferController`)

- `POST /api/roadassistance/requests/{requestId}/offers`  
  Создать предложение помощи по запросу, создаёт чат типа `ASSISTANCE`.
- `POST /api/roadassistance/offers/{offerId}/cancel`  
  Отменить своё предложение.
- `GET /api/roadassistance/requests/{requestId}/offers/my`  
  Получить предложение текущего пользователя по конкретному запросу (или `404`, если его нет).
- `PUT /api/roadassistance/requests/{requestId}/offers/{offerId}/accept`  
  Принять предложение (право имеет автор запроса).
- `POST /api/roadassistance/requests/{requestId}/offers/{offerId}/reject`  
  Отклонить предложение.

#### 5.1.3. Отзывы и модерация (`AssistanceReviewController`)

- `GET /api/roadassistance/requests/{requestId}/review`  
  Черновик отзыва для запроса (в текущей реализации возвращается `404`, `TODO`).
- `POST /api/roadassistance/requests/{requestId}/review`  
  Создать/опубликовать отзыв по запросу.  
  Вход: `revieweeId`, `rating`, `comment` (опционально).
- `GET /api/roadassistance/users/{userId}/reviews`  
  Получить список отзывов по пользователю (пагинация). В текущем коде есть `TODO` на корректное получение `PoputkaUser` по `userId`.
- `POST /api/roadassistance/reports`  
  Создать жалобу на отзыв (`TODO`).
- `GET /api/roadassistance/moderation/low-rated-users`  
  Получить список пользователей с низким рейтингом для модерации (`TODO`/заглушка).
- `PUT /api/roadassistance/moderation/reviews/{reviewId}`  
  Модерировать отзыв, изменить флаги и комментарий модератора. Пересчёт рейтинга выполняется при `isValid=false`.

#### 5.1.4. Профиль помощника (`UserProfileController`)

- `PUT /api/roadassistance/profile`  
  Обновить профиль помощника: `readyToHelp`, `helpRadius`, список `competencyIds`.
- `GET /api/roadassistance/competencies`  
  Получить список доступных компетенций (DTO‑объекты).
- `GET /api/roadassistance/profile/competencies`  
  Получить компетенции текущего авторизованного пользователя.

### 5.2. Планируемые/дорабатываемые эндпоинты

Ниже приведены сценарии, которые описаны в бизнес‑требованиях, но могут быть ещё не полностью реализованы в коде:

- `PUT /api/roadassistance/requests/{requestId}/status` — явное изменение статуса запроса.
- `POST /api/roadassistance/requests/{requestId}/confirm-completion` — подтверждение завершения помощи обеими сторонами.

При реализации этих эндпоинтов их поведение должно соответствовать ограничениям по статусам и времени из разделов выше.

## 6. Интеграции

### 6.1. Чаты

- Используется существующая система чатов с типами `RIDE` и `ASSISTANCE`.
- Для каждого предложения помощи создаётся чат типа `ASSISTANCE`.

### 6.2. Уведомления

Основные типы уведомлений для модуля (строки используются в сервисах):

- `NEW_ASSISTANCE_REQUEST` — новый запрос о помощи (сейчас отправляется всем готовым помощникам; `helpRadius` пока `TODO`);
- `NEW_OFFER` — новое предложение на запрос;
- `OFFER_ACCEPTED` — ваше предложение принято;
- `OFFER_REJECTED` — предложение отклонено;
- `REQUEST_EXPIRING_SOON` — запрос продлён (строка используется для уведомления участников чата);
- `LOW_RATED_USER` — пользователь с низким рейтингом требует модерации.

## 7. Ограничения и правила

- Редактирование сообщений/отмены без автоотзыва: порог **15 минут** указан в логике автоотзыва, но сам auto-review пока не реализован (`TODO`).
- Максимальная длительность запроса: 48 часов.
- Автоматическое истечение без продления: 2 часа.
- Одновременно не более 3 активных запросов на одного пользователя (на уровне бизнес‑логики).
- Пользователь с рейтингом < 3.0 не может создавать новые запросы и предложения (фактически блокируется проверкой `rating >= 3.0` в сервисах).

## 8. Архитектура и безопасность

- Слои:
    - **Controller** — HTTP‑эндпоинты, базовая валидация входа.
    - **Service** — бизнес‑правила, транзакции, статусные переходы.
    - **Repository** — доступ к БД.
    - **Integration** — работа с чатами, уведомлениями и др. внешними системами.
- Транзакции:
    - создание предложения + создание чата;
    - принятие предложения + обновление статусов остальных предложений;
    - создание/публикация отзыва и пересчёт рейтинга.
- Security:
    - используется общая SESSION‑авторизация проекта;
    - проверки владения ресурсами (например, отмена запроса — только автор).

## 9. Мониторинг и логирование

- Ключевые метрики:
    - количество созданных запросов в день;
    - среднее время оказания помощи;
    - процент успешно завершённых запросов;
    - распределение по типам проблем.
- Логировать:
    - создание/изменение запросов и предложений;
    - изменения статусов;
    - действия модерации.
