# Отзывы о попутчике по бронированию (`trip_booking_review`)

Документ описывает модуль отзывов **только для поездок (попутки)**. Модуль взаимопомощи на дороге (`assistance_reviews`, рейтинг помощи) **не меняется** и остаётся отдельным контуром.

Связанная документация:

- бронирование и статусы: `docs/features/trips-booking.md`
- чат бронирования: `docs/features/chat-messages.md`

## 1. Цели

- Пассажир и водитель (владелец поездки) могут оставить **один отзыв друг о друге** в рамках **одной брони** (`booking`).
- Отзыв учитывается в **отдельном рейтинге поездок** (`trip_rating` или аналог на сущности пользователя), **не смешивается** с рейтингом модуля помощи на дороге.
- Отзыв **публично виден и влияет на рейтинг поездок только после модерации**.
- После создания отзыв находится в `DRAFT`; в течение **настраиваемого интервала** (по умолчанию 1 час, см. раздел 5) автор может **редактировать** содержимое; по истечении интервала отзыв **автоматически** переходит в `PENDING_MODERATION` (отдельного действия «submit» нет).

## 2. Сущность и таблица

- **Таблица / сущность:** `trip_booking_review`.
- **Связи:**
  - `booking_id` — FK на `booking(id)` (UUID);
  - `reviewer_id` — пользователь, оставивший отзыв (пассажир брони или владелец поездки);
  - `reviewee_id` — вторая сторона (вычисляется на сервере, клиент **не** передаёт `reviewee` при создании).
- **Содержимое:** оценка `rating` (целое **1..5**), опционально `comment`.
- **Уникальность:** не более одного отзыва от одного участника на одну бронь: уникальный ключ `(booking_id, reviewer_id)`.

## 3. Когда можно создать отзыв

Отзыв разрешён только если выполнены **все** условия:

1. Текущий пользователь — участник брони: либо `passenger_id` брони, либо владелец поездки (`trip.owner_id` для `booking.trip_id`).
2. Статус брони (`booking_status`) — **терминальный**, при котором по продуктовым правилам **уже было взаимодействие по поездке**:
   - **обязательный сценарий:** `COMPLETED`;
3. Статусы вроде `WAITING`, `ACCEPTED`, `IN_PROGRESS`, а также отмены без завершённой поездки (`NO_SHOW`, `REJECTED`, `CANCELLED`, `EXPIRED` и т.п.) **не дают** права на отзыв.

Нельзя оставить отзыв **самому себе** и нельзя при отсутствии участия в брони.

## 4. Статусы отзыва и модерация

Жизненный цикл отзыва (логические статусы, имена можно согласовать с enum в коде):

| Статус               | Смысл                                                                                                  |
|----------------------|--------------------------------------------------------------------------------------------------------|
| `DRAFT`              | Создан автором; до истечения настраиваемого интервала (раздел 5) доступно редактирование, затем **автоматический** переход в `PENDING_MODERATION`. |
| `PENDING_MODERATION` | Отправлен на модерацию, ожидает решения.                                                               |
| `APPROVED`           | Одобрен модератором — **публикуется**, участвует в **рейтинге поездок** и в публичных списках отзывов. |
| `REJECTED`           | Отклонён модератором — **не публикуется**, в агрегат рейтинга поездок **не входит**.                   |

**Публикация:** содержимое отзыва для остальных пользователей и участие в пересчёте `trip_rating` — **только** при статусе `APPROVED`.

После статуса **`REJECTED`**: повторно создать отзыв по **той же брони** тем же **рецензентом** (`reviewer_id`) **нельзя** — одна попытка на пару «бронь + автор отзыва», даже если модерация отклонила текст.

## 5. Редактирование и автоматическая подача на модерацию

- **Интервал в `DRAFT`:** с момента **создания** записи отзыва (`created_datetime` / аналог в аудите) и до истечения настраиваемого периода автор может менять `rating` и `comment`, пока статус остаётся **`DRAFT`**.
- **Настройка:** длительность периода задаётся **одной** конфигурируемой величиной (например свойство Spring Boot `Duration`, имя на усмотрение реализации, значение по умолчанию **1 час**), чтобы её можно было менять без перекомпиляции.
- **Автопереход:** по истечении этого периода система **без участия клиента** переводит отзыв из `DRAFT` в **`PENDING_MODERATION`** (планировщик / отложенная задача / просрочка при следующем тике — деталь реализации). Эндпоинта **`submit` нет**.
- После перехода в `PENDING_MODERATION` (и далее) автор **не** редактирует отзыв через API.
- Редактирование **не допускается** в статусах `APPROVED` и `REJECTED` (окончательное решение модерации).
- Поля ответа `canEdit` / `editableUntil` в `GET .../me` считаются от **того же** настраиваемого интервала (до автоперехода в `PENDING_MODERATION`).

## 6. Рейтинг поездок

- На сущности пользователя хранится **отдельное** поле среднего рейтинга по отзывам попутки (например `trip_rating`), **не** смешивается с полем рейтинга модуля помощи на дороге.
- В расчёт входят только отзывы со статусом **`APPROVED`**, по согласованным правилам (например последние **N** отзывов по дате публикации/одобрения — значение **N** зафиксировать в сервисе пересчёта).
- Пересчёт выполнять при смене статуса на `APPROVED` и при отзыве одобрения, если такая операция предусмотрена.

## 7. API

Модуль выносится в **отдельный контроллер** `TripBookingReviewController`. Все HTTP-маршруты модуля имеют общий префикс:

**`/api/trip-booking-review`**

Дальше в документе пути указаны **относительно этого префикса** (полный путь = префикс + суффикс). Сегмент **`users`** вынесен на уровень этого же префикса, чтобы список отзывов о пользователе жил в том же пространстве имён.

**Аутентификация:** все эндпоинты модуля (включая `GET .../users/{userId}`) доступны **только авторизованным** пользователям; анонимный доступ не предусмотрен.

Рекомендуемая схема сегментов (бронь — UUID, пользователь — числовой id; коллизий в шаблонах Spring MVC нет):

| Назначение                         | Метод и путь (полный URL)                                                                  |
|------------------------------------|--------------------------------------------------------------------------------------------|
| Мой отзыв по брони                 | `GET /api/trip-booking-review/booking/{bookingId}/me`                                      |
| Создать отзыв                      | `POST /api/trip-booking-review/booking/{bookingId}`                                        |
| Редактировать отзыв (только `DRAFT`) | `PATCH /api/trip-booking-review/{reviewId}`                                                |
| Список одобренных отзывов о пользователе | `GET /api/trip-booking-review/users/{userId}`                                          |
| Модерация (очередь / решение)      | подпути вида `/api/trip-booking-review/moderation/...` (конкретные имена — при реализации) |

### 7.1. Мой отзыв по брони (явная структура)

- **URL:** `GET /api/trip-booking-review/booking/{bookingId}/me`
- **Auth:** обязательна; участник брони или владелец поездки (в зависимости от того, кому разрешено «видеть» свою карточку отзыва — минимум автор отзыва).
- **Ответ:** всегда **200** с **явной** обёрткой (без `404` для случая «отзыва ещё нет», чтобы не путать с отсутствием брони).

Рекомендуемая форма JSON:

```json
{
  "hasReview": false
}
```

или

```json
{
  "hasReview": true,
  "review": {
    "id": "uuid или long — как в БД",
    "bookingId": "uuid брони",
    "status": "DRAFT | PENDING_MODERATION | APPROVED | REJECTED",
    "rating": 5,
    "comment": "текст или null",
    "revieweeId": 123,
    "createdAt": 1710000000000,
    "canEdit": true,
    "editableUntil": 1710003600000,
    "moderatorComment": null
  }
}
```

Пояснения полей:

- `hasReview` — есть ли запись отзыва от **текущего** пользователя по данной брони.
- `canEdit` — можно ли сейчас вызывать PATCH (статус `DRAFT` и не истёк настраиваемый интервал, раздел 5).
- `editableUntil` — момент времени (epoch ms), до которого разрешено редактирование автором (= `createdAt` + настроенная длительность); если окно закрыто или статус не `DRAFT`, — `canEdit: false` и при необходимости `editableUntil` в прошлом или `null`.
- `moderatorComment` — причина/комментарий при `REJECTED` (и при необходимости при `APPROVED`), видимость для автора — по политике продукта.

### 7.2. Создание отзыва

- **URL:** `POST /api/trip-booking-review/booking/{bookingId}`
- **Body:** `{ "rating": 1-5, "comment": "опционально" }`
- **Поведение:** создаётся запись в статусе **`DRAFT`**; `reviewee` выставляется сервером. По истечении **настраиваемого интервала** (раздел 5, по умолчанию 1 час) запись **автоматически** переводится в **`PENDING_MODERATION`**. Отдельного API для отправки на модерацию **нет**.

### 7.3. Редактирование

- **URL:** `PATCH /api/trip-booking-review/{reviewId}`
- **Условия:** автор, `canEdit == true`, статус допускает правки (см. раздел 5).

### 7.4. Список отзывов о пользователе

- **URL:** `GET /api/trip-booking-review/users/{userId}`; пагинация — как принято в проекте (например параметры Spring Data).
- **Auth:** **только для авторизованных** пользователей (как и остальные методы модуля).
- **Ответ:** только отзывы со статусом **`APPROVED`**, без персональных данных модерации, с контекстом по продукту (дата, рейтинг, текст, опционально обезличенная поездка).

### 7.5. Модерация

Эндпоинты для роли модератора остаются под тем же префиксом **`/api/trip-booking-review`** (например `.../moderation/pending`, `.../moderation/{reviewId}` — точные пути и тела запросов зафиксировать при реализации). Проверки прав — по принятой в проекте схеме безопасности.

- список отзывов в очереди;
- решение по отзыву: одобрить / отклонить, опционально `moderator_comment`;
- после решения — пересчёт `trip_rating` у `reviewee` при `APPROVED` / при снятии одобрения.

## 8. Ошибки (ориентир)

- **404** — нет брони с таким `bookingId`, нет отзыва с таким `reviewId`, нет пользователя `userId` — в зависимости от эндпоинта; для `GET .../booking/{bookingId}/me` отсутствие **самой брони** или доступа к ней может оформляться как **404** (случай «отзыва ещё нет» по-прежнему **200** с `hasReview: false`, см. раздел 7.1).
- **409 Conflict** — нарушение уникальности `(booking_id, reviewer_id)` (в т.ч. повторное создание после `REJECTED` тем же автором по той же брони), или отзыв в недопустимом статусе брони / отзыва для операции.
- **422** — валидация `rating`/длины `comment`.
- **401** — нет сессии / не аутентифицирован.
- **403** — не участник брони, запрет модерации, доступ к чужим данным и т.п.

## 9. Реализация в кодовой базе

- Отдельный класс контроллера (`TripBookingReviewController`) с `@RequestMapping("/api/trip-booking-review")`; все методы — только для аутентифицированных пользователей.
- Интервал `DRAFT` → автоматический `PENDING_MODERATION`: настройка `poputka.trip-booking-review.draft-to-moderation-delay` (`Duration`, по умолчанию **1 час**), используется и для расчёта `editableUntil` / `canEdit`. Планировщик: `TripBookingReviewDraftScheduler`, период опроса `poputka.trip-booking-review.poll-ms` (по умолчанию 60 с), профиль `test` отключает планировщик.
- Модераторы: `poputka.trip-booking-review.moderator-usernames` — список логинов; пустой список означает, что модерация через API недоступна (403).
- Миграции Liquibase: `2026-04-18-trip-booking-review.yaml` (таблица `trip_booking_review`, колонка `users.trip_rating`).
- Слои: controller → service → repository; DTO и MapStruct — по `AGENT.md`.
- Тесты: `TripBookingReviewControllerTest`, данные не-`users` — **в Java**, не в test Liquibase (см. правила проекта).

## 10. Опциональные расширения

- Сервисное сообщение в чате брони при смене статуса отзыва (например `payload.event = "TRIP_BOOKING_REVIEW_STATUS_CHANGED"`) — по аналогии с `BOOKING_STATUS_CHANGED` в `chat-messages.md`.
- Уведомление автору о результате модерации.
